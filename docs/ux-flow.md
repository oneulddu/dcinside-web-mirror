# UX Flow - 숨터 기본 화면 리디자인

## User Goal

- 사용자가 갤러리를 찾고, 게시글 목록을 훑고, 글을 읽은 뒤 댓글과 관련 글로 이어서 이동한다.
- 화면은 오래 읽어도 피로가 적고, 모바일에서도 주요 조작이 한눈에 들어와야 한다.

## Primary Path

1. 홈에서 갤러리 검색 또는 흥한 갤러리 목록을 본다.
2. 갤러리 행을 눌러 게시판 목록으로 이동한다.
3. 말머리, 전체/베스트 탭, 검색을 필요할 때만 조정한다.
4. 게시글 행을 눌러 읽기 화면으로 이동한다.
5. 본문, 댓글, 다른 게시글을 순서대로 읽고 목록으로 돌아간다.

## Secondary Paths

- 홈 검색어를 초기화하고 흥한 갤러리 순위로 돌아간다.
- 게시판에서 추천 보기와 전체 보기를 오간다.
- 게시판 하단 검색으로 제목, 내용, 글쓴이, 댓글 범위를 바꾼다.
- 최근 방문 화면에서 이전 갤러리로 바로 이동한다.
- 읽기 화면에서 관련 글 더보기를 눌러 다음 글을 이어 본다.
- 테마 전환 버튼으로 어두운 화면과 밝은 화면을 바꾼다.

## Screen States

| State | Required behavior |
|---|---|
| Loading | 게시글 행과 버튼은 active/pressed 상태를 즉시 보여준다. 관련 글 더보기는 버튼 자체가 loading 상태를 표시한다. |
| Empty | 홈, 게시판, 최근, 댓글, 관련 글은 각각 빈 행 문구를 유지한다. |
| Partial | 검색 실패나 흥한 갤러리 로딩 실패는 기존 오류 문구를 행 안에서 표시한다. |
| Success | 목록 행, 메타, 배지, 페이저가 같은 괘선 체계로 정렬된다. |
| Error | 서버 오류 문구는 기존 라우트 메시지를 유지하고 레이아웃이 깨지지 않는다. |
| Disabled | 이전 페이지가 없을 때 페이저는 비활성 스타일과 포인터 비활성을 가진다. |
| Selected | 활성 탭과 선택된 말머리는 `aria-current` 또는 `.active`와 밑줄로 표시한다. |
| Hover/Focus | 모든 링크, 버튼, 입력은 hover와 focus-visible이 시각적으로 보인다. |
| Mobile | 390px에서 검색 폼과 페이저가 줄바꿈되어 가로 스크롤이 없어야 한다. |
| Read | 긴 제목, 넓은 이미지, iframe, 테이블, 디시콘 댓글, 답글 들여쓰기가 컬럼 밖으로 넘치지 않아야 한다. |
| Related exhausted/error | 더 불러올 글이 없거나 실패했을 때 버튼 레이블과 비활성 상태가 명확해야 한다. |

## Information Hierarchy

- 첫 번째: 브랜드, 현재 위치, 전체/베스트/최근 탭.
- 두 번째: 검색 또는 갤러리/게시판 제목, 갱신 시각, 추천 전환.
- 세 번째: 목록 제목과 댓글 수.
- 네 번째: 작성자, 말머리, 시간, 추천 수 같은 메타.
- 필요할 때만: 페이지 이동, 검색 범위, 관련 글 더보기.

## Interaction Contract

- 브랜드: 홈으로 이동한다.
- 테마 버튼: 현재 테마를 반대로 바꾸고 로컬 저장소에 저장한다.
- 상단 탭: 전체, 베스트, 최근 방문 화면으로 이동한다.
- 홈 검색 입력: 갤러리명 또는 board id로 검색한다.
- 검색 초기화: 홈 기본 흥한 갤러리 목록으로 돌아간다.
- 게시글 행: 읽기 화면으로 이동하고 읽음 상태를 저장한다.
- 말머리 탭: 해당 분류의 1페이지로 이동한다.
- 게시판 검색: 선택된 범위와 검색어로 1페이지 검색 결과를 연다.
- 검색 해제: 검색 조건 없이 같은 게시판으로 돌아간다.
- 관련 글 더보기: JSON 엔드포인트에서 다음 관련 글을 추가한다.
- 관련 글 더보기 실패: 버튼을 다시 누를 수 있게 유지하고 실패 문구를 버튼 레이블에 표시한다.
- 맨 위로: 읽기 화면 상단으로 부드럽게 이동한다.

## Risk Checks

- 검색 폼이 모바일에서 버튼과 겹치지 않아야 한다.
- 제목이 길어도 댓글 수와 아이콘이 밀려 화면 밖으로 나가지 않아야 한다.
- 다크 모드에서 favicon 표식, 링크, 테두리 대비가 유지되어야 한다.
- 읽기 본문 이미지와 iframe이 컬럼 밖으로 넘치지 않아야 한다.
- 비활성 페이저가 링크처럼 보이면 안 된다.
- 장식성 UI가 정보 밀도를 낮추면 제거한다.
- 목록 행 제목은 헤딩이 아니라 일반 텍스트 역할이어야 한다. 섹션 제목만 헤딩으로 남긴다.
- 외부 이동 갤러리는 행을 누르기 전에 외부 새 탭 이동임을 알 수 있어야 한다.
- 테마 토글은 첫 페인트와 JS 실패 상황에서도 현재 테마를 거짓으로 말하지 않아야 한다.
- 관련 글 더보기로 추가되는 행은 서버 렌더링 행과 같은 구조와 클래스를 써야 한다.

## Acceptance Checks

- `/`, `/board?board=airforce&page=1`, `/read?board=airforce&pid=<실제글>`, `/recent`가 390px, 768px, 1280px에서 가로 스크롤 없이 보인다.
- 홈 검색 폼, 상단 탭, 페이저, 게시판 검색 폼이 keyboard focus-visible을 가진다.
- 읽기 화면에서 긴 제목, 본문 이미지, 댓글, 관련 글 더보기가 깨지지 않는다.
- 읽음 상태, active/pressed 상태, disabled 상태가 양쪽 테마에서 구분된다.
- 라이트와 다크 테마가 모두 토큰 색만 사용한다.
- Superloopy anti-slop 점검에서 em dash, 보라 그라디언트, 가짜 스크린샷, 미등록 색상 사용이 없다.
- Fable 비평과 구현 시도가 문서와 명령 결과로 남는다.

## Current Task Addendum - Embedded Content Rebuild (실제 비율 기반)

이전 임베드 높이 개선(9:16 고정)은 가로 영상에서 문제를 만들었다. 실측 결과
`/read?board=baseball_new11&pid=20969362`의 DCInside 동영상 임베드가 332x590 세로 박스로
강제되어 가로 영상이 거대한 검은 박스 가운데 얇은 띠로만 보인다. 임베드 UX를 실제
콘텐츠 비율 기반으로 다시 만든다.

- DCInside 동영상 임베드는 같은 출처 프록시 플레이어(`/movie`)가 영상 메타데이터를
  `postMessage`로 부모에 알리고, 읽기 화면이 iframe 비율을 실제 영상 비율로 맞춘다.
- 가로 영상은 본문 컬럼 전체 폭을 쓰고, 세로 영상은 폭을 제한해 중앙 정렬하며
  화면 높이를 넘지 않는다.
- 메타데이터 수신 전과 JS 실패 시 기본 프레임은 16:9다. 세로 강제(9:16)는 폐기한다.
- 동영상 로드 실패(삭제 등) 시 임베드는 거대한 빈 박스 대신 낮은 안내 박스로 줄어든다.
- YouTube 임베드는 16:9를 유지한다. 재생 불가(소유자 임베드 차단)는 YouTube가 자체
  안내를 렌더링하므로 프레임 크기만 유지한다.
- 투표 임베드는 항목을 읽고 선택할 수 있는 최소 높이를 유지한다.
- 어떤 셀렉터에도 매칭되지 않는 iframe도 브라우저 기본 150px로 납작해지지 않아야 한다.
- 모든 임베드는 모바일(390px)에서 본문 컬럼 밖으로 넘치거나 가로 스크롤을 만들면 안 된다.
- 키보드로 임베드에 초점이 들어갔을 때 초점 표시가 장식용 외곽선에 묻히면 안 된다.
- 크기 전환은 즉시 적용하고 높이 애니메이션을 쓰지 않는다.
