# 리팩터링 로드맵

이 문서는 현재 코드 상태를 기준으로, 남은 구조 개선 작업을 우선순위별로 정리한 작업 목록입니다.  
각 작업은 한 번에 크게 갈아엎기보다, 테스트가 통과하는 작은 단위로 나누어 진행합니다.

## 진행 상태 요약

| 우선순위 | 항목 | 상태 | 비고 |
|---|---|---|---|
| 높음 | `routes.py` 분리 | 완료 | 미디어 프록시, HTML 정리, 흥갤/검색, 최근 방문, 비동기 브리지 분리 완료 |
| 높음 | `dc_api.py` 리팩터링 | 완료 | `to_int()` 공통화, 목록/문서 파싱 헬퍼 분리, 인라인 테스트 제거 및 pytest 보강 완료 |
| 중간 | 동기-비동기 전략 정리 | 완료 | `asgiref` 기반 하이브리드 브리지 도입, running-loop fallback 유지 |
| 중간 | 다크/라이트 테마 토글 | 완료 | 버튼 동작 및 화이트모드 초기 깜빡임 보정 완료 |
| 낮음 | `requirements.txt` 분리 | 완료 | `requirements.txt` / `requirements-dev.txt` 분리 완료 |
| 낮음 | `legacy/` 정리 | 완료 | 런타임 미사용 확인, 보관 기록 문서화. 삭제/추적 해제는 별도 커밋 권장 |

## 1단계: `routes.py` 분리

### 목표

`app/routes.py`가 라우팅과 응답 조립에 집중하도록 만들고, 보조 로직은 역할별 모듈로 분리합니다.

현재 `routes.py`에는 다음 역할이 섞여 있습니다.

- 라우트 핸들러
- 미디어 프록시
- HTML 새니타이저
- 본문 이미지 URL 재작성
- 흥한 갤러리 크롤링/캐시
- 갤러리 검색
- 최근 방문 쿠키/서버 캐시
- 비동기 실행 브리지

### 분리 후보

| 새 파일 | 옮길 책임 |
|---|---|
| `app/services/media_proxy.py` | 미디어 URL 검증, 허용 호스트 검사, 리다이렉트 처리, 응답 크기 제한, 콘텐츠 타입 검사 |
| `app/services/html_sanitizer.py` | HTML 태그/속성 허용 목록, HTML 정리, 본문 이미지 프록시 URL 재작성 |
| `app/services/heung.py` | 흥한 갤러리 수집, 파일 캐시, 메모리 캐시, 갤러리 검색 |
| `app/services/recent.py` | 최근 방문 쿠키 파싱/저장, 서버 캐시, 최근 방문 포맷 |
| `app/services/async_bridge.py` | Flask 동기 라우트에서 비동기 서비스 호출 |

### 진행 순서

1. 미디어 프록시 로직 분리 ✅ 완료
2. HTML 새니타이저/이미지 재작성 로직 분리 ✅ 완료
3. 흥한 갤러리/갤러리 검색 로직 분리 ✅ 완료
4. 최근 방문 로직 분리 ✅ 완료
5. `_run_async()`를 별도 모듈로 이동 ✅ 완료
6. 기존 테스트 통과 확인 ✅ 완료

### 주의점

- `/media`의 SSRF 방어가 약해지면 안 됩니다.
- HTML 정리 허용 목록이 바뀌면 게시글 렌더링과 XSS 방어에 영향이 큽니다.
- 최근 방문 쿠키 형식은 기존 사용자 쿠키와 호환되어야 합니다.
- `routes.py`의 공개 라우트 URL은 바꾸지 않습니다.

## 2단계: `dc_api.py` 리팩터링

### 목표

긴 메서드와 중복 헬퍼를 줄여, DCinside HTML 구조 변경에 대응하기 쉽게 만듭니다.

현재 확인된 문제:

- `to_int()`가 여러 곳에 로컬 함수로 중복 정의되어 있음
- `API.board()`가 약 250줄로 길고 모바일/PC 파싱이 한 메서드에 섞여 있음
- `API.document()`도 길고 제목/작성자/본문/댓글/이미지 파싱 책임이 섞여 있음
- 파일 하단에 `unittest` 기반 인라인 테스트가 남아 있음

### 분리 후보

| 대상 | 권장 조치 |
|---|---|
| 숫자 파싱 헬퍼 | 모듈 상단 공통 함수로 통합 |
| 모바일 목록 행 파싱 | 별도 메서드로 분리 |
| PC 목록 행 파싱 | 별도 메서드로 분리 |
| `board()` 페이지 순회 | 파싱 함수 호출 중심으로 단순화 |
| 문서 제목/작성자 파싱 | 별도 메서드로 분리 |
| 문서 본문/이미지 파싱 | 별도 메서드로 분리 |
| 인라인 테스트 | `tests/` 아래 테스트 파일로 이동 |

### 진행 순서

1. `to_int()` 공통화 ✅ 완료
2. `board()` 내부 모바일/PC 행 파싱 분리 ✅ 완료
3. `document()` 내부 제목/메타/본문 파싱 분리 ✅ 완료
4. 인라인 테스트를 `tests/`로 이동하거나 삭제 가능 여부 확인 후 정리 ✅ 완료
5. 기존 테스트 통과 확인 ✅ 완료

### 주의점

- DCinside HTML은 자주 바뀌므로 파서 변경은 작게 나누어야 합니다.
- 목록 파싱 결과 필드(`id`, `title`, `author`, `author_id`, `comment_count`, `voteup_count`)가 기존과 같아야 합니다.
- 모바일/PC 폴백 URL 우선순위를 바꾸지 않습니다.

## 3단계: 동기-비동기 전략 정리

### 현재 상태

완료된 작업입니다.

Flask 라우트는 동기 함수이고, DCinside 호출은 `aiohttp` 기반 비동기 함수입니다.  
현재는 `app/services/async_bridge.py`의 `run_async()`가 다음 방식으로 연결합니다.

- 이벤트 루프가 없고 `asgiref`가 있으면 `async_to_sync` 사용
- 이벤트 루프가 없고 `asgiref`가 없으면 `asyncio.run()` fallback
- 이미 이벤트 루프가 있으면 `ThreadPoolExecutor`에서 `asyncio.run()` 실행
- fallback 스레드 수는 `MIRROR_ASYNC_BRIDGE_WORKERS`로 조절 가능

### 검토 방향

| 선택지 | 장점 | 단점 |
|---|---|---|
| 현재 구조 유지 | 변경 적음 | 요청마다 이벤트 루프 생성 비용, 구조가 어색함 |
| `asgiref.sync.async_to_sync` 도입 | 표준화된 sync/async 브리지 | 의존성 추가, 성능 개선은 제한적 |
| ASGI/Quart 전환 | 구조적으로 가장 자연스러움 | 영향 범위 큼, 배포/테스트 수정 필요 |

### 진행 결과

1. `routes.py` 분리 이후 `_run_async()`를 독립 모듈로 이동 ✅ 완료
2. `asgiref` 도입 여부를 작은 변경으로 검토 ✅ 완료
3. `asgiref` 기반 하이브리드 브리지와 running-loop fallback 테스트 추가 ✅ 완료

### 남은 선택 작업

비동기 호출이 더 많아지거나 요청 처리량 문제가 커질 경우 ASGI/Quart 전환은 별도 큰 작업으로 다시 검토합니다.

## 4단계: 테마 토글 정리

### 현재 상태

완료된 작업입니다.

- 테마 토글 버튼 동작
- `localStorage` 기반 라이트/다크 테마 저장
- 새 페이지 이동 시 테마 유지
- CSS 로드 전 초기 테마 적용으로 화이트모드 검은 화면 깜빡임 완화

### 남은 선택 작업

현재는 라이트모드 세부 스타일 일부를 JS에서 `<style>`로 주입합니다.  
나중에 정리한다면 `main.css`의 `html[data-theme='light']` 블록으로 옮겨 구조를 단순화할 수 있습니다.

## 6단계: `legacy/` 정리

### 현재 상태

완료된 작업입니다.

`legacy/`는 런타임에서 사용하지 않는 보관용 코드입니다.  
`legacy/README.md`에도 현재 서비스는 `app/` 하위 코드만 사용한다고 적혀 있습니다.
이번 작업에서는 `docs/legacy-inventory.md`에 보관 범위와 제거 절차를 정리했습니다.

### 선택지

| 선택지 | 장점 | 단점 |
|---|---|---|
| 현재처럼 보관 | 과거 코드 참고 가능 | 저장소가 무거워지고 검색 노이즈 발생 |
| 별도 브랜치/태그로 이동 | main 정리 | 과거 코드 접근이 약간 번거로움 |
| 압축 아카이브로 보관 | 파일 수 감소 | diff 추적 어려움 |
| 삭제 | 가장 깔끔함 | 복구는 Git 히스토리에 의존 |

### 결정

현재 리팩터링 변경과 대량 삭제를 섞지 않기 위해 당장 삭제하지 않습니다.  
삭제하거나 Git 추적을 해제할 경우 `docs/legacy-inventory.md`의 절차에 따라 별도 커밋/PR로 처리합니다.

## 작업 원칙

- 한 번에 하나의 큰 주제만 처리합니다.
- 각 단계마다 테스트를 실행합니다.
- 공개 URL과 기존 쿼리 파라미터는 유지합니다.
- 보안 관련 로직은 분리하더라도 동작을 약화시키지 않습니다.
- 리팩터링 중 기능 변경이 필요한 경우 별도 작업으로 분리합니다.
